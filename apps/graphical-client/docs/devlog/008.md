# 008 - Gameplay Surface Integration

## Status

Planned.

## Context

`006` established the 3D table foundation. `007` finishes the fixed HUD
frame. The next useful slice is to make that combined presentation reflect the
actual authoritative match flow.

This needs to stay aligned with the current game-server RPC contract instead
of overcommitting to rule phrasing that may still evolve in higher-level
design docs. The goal is not a final polished board. The goal is to consume
the existing authoritative game events cleanly on the new surface.

## Goal

Deliver the first gameplay-integrated 3D board surface that:

- consumes authoritative board and turn state from the server
- updates pawn positions from match events instead of default spawn only
- renders landing and pending-action state clearly enough to play
- keeps reconnect and snapshot installation behavior correct

## Proposed Shape

- keep the server as the only source of truth for movement and turn flow
- treat the current RPC contract as canonical for this milestone
- wire the existing board/pawn foundation into event-driven updates
- let the HUD shell absorb current-player and log updates
- defer visual polish until the basic turn loop is playable on the 3D board

## In Scope

- consume `rpc_board_state(...)` and reconnect snapshot board state in the
  gameplay root
- consume `rpc_turn_started(...)`, `rpc_pawn_moved(...)`, and
  `rpc_tile_landed(...)`
- move pawns from authoritative tile positions during play
- reflect current-player and pending-action state in the gameplay UI
- support the current roll / buy / end-turn / pay-toll flow on the 3D surface
- keep tile ownership and tile metadata readable from authoritative state

## Out Of Scope

- final animation and camera treatment
- broad rule redesign to match aspirational docs
- new mechanics beyond the current server contract
- final art pass for ownership, incidents, or endgame presentation

## Acceptance Criteria

- a connected client can enter the game scene and see the authoritative board
  state
- pawns move in response to authoritative movement events
- the active player and pending action are readable on the gameplay surface
- the current buy / pass / pay-toll flow remains usable on the new board
- reconnect/sync restores the board without desynchronizing live events

## Notes On Implementation

This milestone should anchor itself to the implemented RPC behavior documented
in `apps/game-server/docs/RPC_API.md`, especially:

- `rpc_board_state(...)`
- `rpc_turn_started(...)`
- `rpc_pawn_moved(...)`
- `rpc_tile_landed(...)`
- reconnect snapshot installation and sync completion behavior

If design language in broader rules notes changes again, this doc should still
describe the integration work in terms of the authoritative runtime contract.

## Manual Validation Checklist

1. Start a match with at least two players and enter the game scene.
2. Confirm the initial board state matches the authoritative snapshot.
3. Roll and confirm the active pawn moves to the authoritative destination tile.
4. Land on an unowned tile and confirm the buy/pass state is readable.
5. Land on an owned tile and confirm the pay-toll state is readable.
6. Reconnect during a match and confirm the board state restores correctly.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- richer landing and reward feedback
- ownership stack visuals and tile mutation polish
- movement/camera animation polish
- endgame and economy presentation cleanup
