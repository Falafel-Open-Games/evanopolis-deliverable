# 007 - HUD Foundation

## Status

Planned.

## Context

`006` established the 3D table foundation and the pawn runtime. The next
useful slice is the fixed HUD shell that sits above that table:

- the top bar already exists and follows the approved mockup
- the remaining fixed panels should match that same frame
- the HUD should become a stable layout system before gameplay state binding
  gets broader

This is a presentation milestone, not a gameplay-logic milestone. The goal is
to finish the stable HUD frame so later runtime state can land in a clear,
consistent place.

## Goal

Deliver the fixed HUD foundation that:

- keeps the approved top bar in the game scene
- adds a player list panel
- adds an event log panel
- treats the three fixed panels as one consistent HUD system
- leaves room for later gameplay bindings without forcing a redesign

## Proposed Shape

- keep the top bar as the dominant header panel
- place the player list and event log as stable companion panels
- make the fixed panels feel like one intentional frame, not three unrelated
  widgets
- wire the panels to currently available identity and match data where that is
  already safe
- avoid gameplay-rule work until the panel structure is settled

## In Scope

- top bar integration in the active game scene
- player list panel layout and presentation
- event log panel layout and presentation
- shared HUD composition rules for the fixed panels
- lightweight data binding from the current available player/session state
- preserving readability at the current desktop/mobile target sizes

## Out Of Scope

- authoritative turn resolution
- pawn movement or landing behavior
- buy / pass / pay-toll interaction logic
- board-state reconciliation or reconnect handling
- final visual polish on gameplay transitions
- any redesign of the approved top bar treatment

## Acceptance Criteria

- the game scene shows a stable fixed HUD frame
- the top bar remains visible and consistent with the approved mockup
- the player list is visible and readable
- the event log is visible and readable
- the HUD feels like one coherent system rather than a set of ad hoc panels
- the HUD can accept future gameplay bindings without a structural rewrite

## Notes On Implementation

The existing HUD pieces provide the start of this milestone:

- `TopBar` is already implemented as the main header panel
- `GameHudIntroController` already manages the intro transition into the top bar
- `TurnHud` already exists as the HUD layer entry point

This milestone should extend that fixed-panel frame rather than replacing it.
If the player list or event log need additional data plumbing, keep that
plumbing local to the HUD layer for now.

## Manual Validation Checklist

1. Enter the game scene after the waiting room.
2. Confirm the top bar appears and matches the approved frame.
3. Confirm the player list panel is visible and readable.
4. Confirm the event log panel is visible and readable.
5. Confirm the HUD does not overlap the board in a way that harms playability.
6. Confirm the HUD still looks like one system on the target viewport.

## Deferred Follow-Up

The next milestone after this one should be gameplay-surface integration on
top of the completed HUD shell.

That work will live in `008.md`.
