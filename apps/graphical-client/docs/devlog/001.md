# 001 - Web Boot And Launch Bridge

## Status

Done.

## Context

This graphical client is being rebuilt as a greenfield Godot 4.6 project.

The current public wrapper flow already performs:

- auth
- room creation and invite lookup
- pay-to-enter
- payment verification
- navigation into `/launch.html`

At launch time, the wrapper has a payload with:

- `token`
- `gameId`
- `gameServerUrl`
- `playerAddress`

This first task is not about gameplay yet. It is about establishing a clean,
documented, minimal boot path for the new Godot client when embedded in the
wrapper launch iframe.

## Goal

Deliver a minimal Godot web client that:

- exports to Web/HTML5
- can be embedded by the wrapper launch page
- has the hooks needed to receive launch data from the wrapper
- documents the contract clearly in source and in this devlog entry

## Proposed Shape

- Godot main scene: `scenes/app/app_boot.tscn`
- Godot boot script: `scripts/app/app_boot.gd`
- wrapper launch page stops using iframe placeholder HTML and mounts the real
  exported client iframe
- parent and child communicate through an explicit iframe message contract

## Contract Direction

Use `postMessage` for the wrapper-to-iframe handoff.

Do not have the Godot client read wrapper `sessionStorage` directly.

The wrapper may continue using session storage internally, but the embedded
client should receive launch data through a deliberate parent/child boundary.

### Initial Message Contract

Use a broad host/client protocol envelope so the same channel can support
future non-launch messages such as session refresh or wrapper-owned state
changes.

Child iframe to parent:

```json
{
  "protocol": "open-game-host",
  "version": 1,
  "type": "client_ready"
}
```

Parent to child:

```json
{
  "protocol": "open-game-host",
  "version": 1,
  "type": "launch_payload",
  "payload": {
    "token": "<jwt>",
    "gameId": "<room-id>",
    "gameServerUrl": "wss://...",
    "playerAddress": "0x..."
  }
}
```

## In Scope

- minimal greenfield Godot project structure
- `app_boot.tscn` set as the project main scene
- local Godot web export path documented and working
- wrapper launch integration updated to embed the exported client
- iframe ready/launch-payload handshake implemented
- Godot boot scene receives and validates launch payload
- minimal boot UI states for:
  - waiting for host
  - payload received
  - payload missing or invalid
- a large explanatory comment block in `app_boot.gd`

## Out Of Scope

- actual game-server connection
- auth RPC call implementation
- join RPC call implementation
- waiting room or lobby behavior
- gameplay scenes
- reconnect logic
- committed export artifacts
- full automated integration tests across wrapper and iframe

## Acceptance Criteria

- the Godot project exports to Web successfully
- `/launch.html` embeds the exported Godot client instead of placeholder HTML
- the embedded client announces readiness to the parent
- the parent sends launch data to the iframe
- the Godot boot scene receives and parses the payload
- the Godot UI shows a safe confirmation state without rendering the raw token
- missing payload produces a clear error or fallback state
- the source files are readable enough to serve as working documentation

## Notes On Documentation And Tests

For this first chunk, prioritize:

- strong source comments in `app_boot.gd`
- explicit message names and payload shape
- a short manual validation checklist

Automated tests are still desirable, but they are not required for this first
milestone if they would make the chunk materially larger or distract from
stabilizing the contract.

## Manual Validation Checklist

1. Export the Godot project for Web.
2. Open the wrapper flow locally.
3. Complete the existing payment verification path.
4. Launch into `/launch.html`.
5. Confirm the real iframe loads.
6. Confirm the child sends `open-game-host` / `client_ready`.
7. Confirm the parent replies with `open-game-host` / `launch_payload`.
8. Confirm Godot shows that launch data was received.
9. Confirm the JWT is not displayed raw in the UI.
10. Confirm missing launch data shows a clear failure state.
