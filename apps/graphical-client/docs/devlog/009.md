# 009 - Gameplay Action UI

## Status

Planned.

## Context

`008` connected the 3D gameplay surface to the authoritative board, turn,
movement, and landing events. The remaining usability gap is the player action
surface for pending landing decisions.

The server already exposes the current action contract. This milestone should
make those decisions visible and directly usable in the graphical client
without changing the game rules.

## Goal

Deliver the graphical action UI for pending tile decisions:

- buy an unowned property
- pass / end turn when buying is optional
- pay toll when landing on another player's property
- keep roll and end-turn controls coherent with pending-action state

## Proposed Shape

- keep `SessionCheck` as the source of truth for action availability
- expose pending-action metadata to the game root in a small, explicit shape
- add a focused action panel near the existing turn controls
- keep the UI textual and clear before adding visual polish
- route all actions through the existing server RPCs

## In Scope

- consume `action_required` from `rpc_tile_landed(...)`
- surface `buy_or_end_turn` with buy and pass controls
- surface `pay_toll` with pay-toll control
- display the relevant tile label, price, owner, or toll amount when available
- disable controls while requests are pending
- preserve reconnect behavior when a snapshot includes a pending action

## Out Of Scope

- new gameplay mechanics
- redesigning the HUD frame
- ownership stack visuals
- final animation and camera treatment
- economy redesign or copy polish beyond what is needed for clarity

## Acceptance Criteria

- landing on an unowned property lets the current player buy or pass from the
  graphical client
- landing on an owned property lets the current player pay toll from the
  graphical client
- non-current players can see pending action context without actionable
  controls
- controls disable while the request is in flight and recover on rejection
- reconnect/sync restores any pending action UI correctly

## Notes On Implementation

This milestone should stay aligned with:

- `rpc_tile_landed(seq, tile_index, tile_type, city, owner_index, toll_due,
  buy_price, action_required)`
- `rpc_buy_property(game_id, player_id, tile_index)`
- `rpc_pay_toll(game_id, player_id)`
- `rpc_end_turn(game_id, player_id)`
- snapshot `pending_action` metadata

Avoid duplicating rule decisions in the UI. The UI should present and submit
the action state the server already exposes.

## Manual Validation Checklist

1. Start a two-player match.
2. Land on an unowned property and buy it from the graphical client.
3. Land on an unowned property and pass / end turn from the graphical client.
4. Land on another player's property and pay toll from the graphical client.
5. Attempt the same pending-action views as a non-current player.
6. Reconnect while a pending action is active and confirm the UI restores.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- richer landing and reward feedback
- ownership stack visuals and tile mutation polish
- movement/camera animation polish
- endgame and economy presentation cleanup
