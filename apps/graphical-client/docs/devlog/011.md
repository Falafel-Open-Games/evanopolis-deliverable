# 011 - Center-Board Dice Presentation

## Status

Planned.

## Context

`010` completed the first authoritative energy-allocation loop. The next
missing piece in the gameplay surface is readable feedback for the actual dice
result that drives pawn movement.

The current graphical client already lets the active player request a roll and
the server already broadcasts `rpc_dice_rolled(seq, die_1, die_2, total)`.
Today that result is only visible in the event log. The board itself does not
yet show the rolled dice in a way that feels tied to the central play surface.

## Goal

Deliver the first center-board dice presentation in the graphical client so
that:

- the authoritative dice result appears in the middle of the board
- both die faces are readable without relying on the event log
- the presentation feels like part of the 3D table instead of HUD-only text
- the result remains grounded in the server-authoritative roll event

## Proposed Shape

- keep the server as the only source of truth for die values
- treat dice presentation as a client-side view of `rpc_dice_rolled(...)`, not
  as a physics or randomness system
- place the dice near the visual center of the table where both players can
  read them clearly
- support a short presentational animation if it improves readability, but keep
  the outcome deterministic and event-driven
- keep the implementation modular so later polish can replace the first visual
  treatment without rewriting gameplay state flow

## In Scope

- consume `rpc_dice_rolled(...)` into a dedicated gameplay presentation path
- add a center-board dice presentation node or scene to the gameplay root
- show both die values and the rolled total clearly enough for live play
- optionally animate dice arrival, reveal, bounce, spin, or settle behavior
  after the authoritative event arrives
- hide or reset stale dice state when appropriate for turn progression
- preserve reconnect behavior without inventing fake rolls that the server did
  not send

## Out Of Scope

- server-side rule changes to rolling or movement
- client-side random physics that can disagree with authoritative values
- camera-system redesign just to feature the dice
- final VFX, audio, particles, or cinematic polish
- historical roll timelines beyond current-turn readability

## Acceptance Criteria

- when the server emits `rpc_dice_rolled(...)`, the graphical client shows two
  readable dice in the center of the board
- the dice faces match the authoritative `die_1` and `die_2` values
- the total is readable on the board surface without checking the event log
- the presentation does not block the existing authoritative movement flow
- reconnect/sync does not fabricate or replay an incorrect dice result

## Notes On Implementation

This milestone should stay anchored to the existing runtime contract:

- `rpc_roll_dice(game_id, player_id)`
- `rpc_dice_rolled(seq, die_1, die_2, total)`
- `rpc_pawn_moved(...)`
- reconnect snapshot installation in `SessionCheck`

The key design choice is representational, not mechanical. The client should
present the server's chosen values, not simulate a local roll and hope it
matches.

For a first slice, a non-physics animation is acceptable if it is easier to
control and keeps the dice readable. A later milestone can replace the first
animation with richer 3D treatment once the presentation path is stable.

## Manual Validation Checklist

1. Start a match and enter the graphical gameplay scene.
2. Roll dice as the active player and confirm both die faces appear in the
   middle of the board.
3. Confirm the shown total matches the sum of the two authoritative die faces.
4. Confirm pawn movement still follows the normal server-driven flow after the
   dice presentation appears.
5. Watch another player's roll and confirm the dice remain readable to a
   non-active observer.
6. Reconnect during a match and confirm the client does not invent a stale or
   incorrect dice presentation.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- richer dice animation and table interaction
- sound design and impact polish
- camera framing or emphasis during roll reveal
- stronger integration between dice presentation and pawn movement timing
- roll history or recent-result UI if live play needs it
