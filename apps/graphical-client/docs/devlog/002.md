# 002 - Session Check Bridge

## Status

Done.

## Context

The current graphical client already has a clean boot bridge:

- the wrapper sends launch data through `open-game-host`
- the Godot boot scene receives and validates the launch payload
- the launch payload is now represented as a typed runtime model
- the web bridge and command-line launch path are separated from the boot scene

That is enough to prove the launch contract, but the player experience still
stops at boot confirmation. The next useful step is to make the client do one
real thing with the payload: use it to enter the server handshake and verify
that the session is actually usable.

This is still not the waiting room. It is the bridge between launch and the
first real server-backed player state.

## Goal

Deliver a minimal Godot session-check stage that:

- consumes the typed launch payload from `AppBoot`
- opens the multiplayer connection to `gameServerUrl`
- sends the token as the first auth RPC
- waits for `rpc_auth_ok(player_id, exp)` or `rpc_auth_error(reason)`
- sends `rpc_join(game_id, player_id)` after auth success
- waits for `rpc_join_accepted(seq, player_id, player_index, last_seq)`
- optionally requests sync if the match flow requires it
- reports a small set of readable connection states
- fails clearly when the session cannot be reached or is not valid
- retries the initial transport handshake once on transient connection failure

## Proposed Shape

- `AppBoot` remains the launch entrypoint and payload orchestrator
- a new session-check scene or controller handles the first server-backed step
- the session-check layer should use the existing RPC contract rather than
  introducing a separate auth protocol in Godot
- the session-check layer should be reusable for later reconnect and token
  refresh work
- the player should land on a neutral pre-waiting-room state only after the
  session check succeeds

## In Scope

- connect to `gameServerUrl`
- establish the multiplayer transport
- send `rpc_auth(token)` as the first application-level message
- verify that the server returns `rpc_auth_ok(...)`
- send `rpc_join(game_id, player_id)` after auth success
- verify that the server returns `rpc_join_accepted(...)`
- optionally request `rpc_sync_request(...)` if the match flow needs an
  immediate authoritative snapshot
- display clear boot-to-session states such as:
  - connecting
  - verifying
  - ready
  - failed
- preserve the launch payload as the source of truth for the server handshake
- keep the boot bridge and the session-check code separated
- retry the initial transport handshake once for transient failures

## Out Of Scope

- waiting room UI
- gameplay scenes
- full reconnect persistence
- token refresh implementation
- player turn logic
- real match interaction
- broad UI redesign
- payment collection in the Godot client

## Acceptance Criteria

- a valid launch payload can proceed past boot into a real session check
- invalid token or invalid room/session fails with a clear state
- a reachable server produces a neutral ready state instead of stopping at boot
- the handshake path visibly matches the game-server RPC contract
- the code path is still readable enough to explain the player flow without a
  separate design document

## Notes On Implementation

The main risk in this chunk is letting session verification sprawl into a
general-purpose networking layer too early. Keep the first version narrow:

- one launch payload in
- one auth RPC
- one join RPC
- one optional sync handoff
- one result into the next player state

This is also the right place to start using the later reusable bridge for
session refresh, but only as a contract shape, not as a fully implemented
feature.

## Manual Validation Checklist

1. Start the wrapper flow and reach `/launch.html`.
2. Confirm the Godot client receives a valid launch payload.
3. Confirm the client connects to the game server.
4. Confirm the client sends `rpc_auth(token)` first.
5. Confirm the server responds with `rpc_auth_ok(...)` and the client sends
   `rpc_join(game_id, player_id)`.
6. Confirm a valid session reaches the ready state.
7. Confirm a transient connection failure retries once before failing.
8. Confirm a bad token or room fails with a clear message.
9. Confirm the code path does not enter a waiting room yet.
