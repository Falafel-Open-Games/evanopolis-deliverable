# 003 - Waiting Room Entry

## Status

Planned.

## Context

The current graphical client can already complete the first real server-backed
session handshake:

- receive launch payload from the wrapper or command-line arguments
- connect to the game server
- send `rpc_auth(token)`
- send `rpc_join(game_id, player_id)`
- request `rpc_sync_request(...)`
- reach a neutral connected state after `rpc_sync_complete(...)`

That proves the client can enter the authoritative session path, but the player
still stops at a generic status card. The next useful step is to turn that
connected state into an actual pre-match waiting room.

This is still not gameplay. It is the first player-facing scene that consumes
authoritative match/session data and lets the player manually mark ready when
the match flow requires it.

## Goal

Deliver a minimal waiting-room stage that:

- starts from the existing connected session state
- consumes the authoritative sync snapshot from the game server
- shows a basic waiting-room view with room and player information
- reflects ready-state updates from the server
- lets the local player trigger `rpc_player_ready(game_id, player_id)` manually
- transitions out of waiting room when the server broadcasts `rpc_game_started`

## Proposed Shape

- keep `AppBoot` as the launch bridge
- keep `SessionCheck` as the auth/join/sync gate
- add a small waiting-room controller and scene after successful sync
- treat the waiting room as the first consumer of `rpc_state_snapshot(...)`
  and match-lifecycle broadcasts such as:
  - `rpc_player_joined(...)`
  - `rpc_player_ready_state(...)`
  - `rpc_game_started(...)`
- keep the waiting-room UI intentionally narrow and readable rather than trying
  to design the full in-game presentation already

## In Scope

- parse and hold the parts of the sync snapshot needed for pre-match display
- show a waiting-room scene instead of the generic "session ready" card
- display local player identity and room/game id
- display known joined players and their ready states
- reflect later `rpc_player_joined(...)` broadcasts after the initial snapshot
- reflect `rpc_player_ready_state(...)` broadcasts
- add a manual local ready action that sends `rpc_player_ready(...)`
- disable or hide the ready action once the local player is already ready
- react to `rpc_game_started(...)` by leaving the waiting room and entering the
  next placeholder player state
- keep the implementation compatible with the current RPC contract and the
  existing text-client behavior

## Out Of Scope

- actual gameplay board rendering
- turn controls such as roll/end-turn/buy/toll actions
- reconnect queueing of live events during snapshot install beyond what is
  already required for the current sync contract
- broad visual polish or final production UI
- payment verification changes in the game server
- token refresh flow
- full match-state modeling for every gameplay RPC

## Acceptance Criteria

- after successful auth/join/sync, the client enters a waiting-room scene
- the waiting room shows meaningful authoritative session information instead of
  only a generic connection status
- ready-state changes are visible when the server broadcasts them
- the local player can manually send `rpc_player_ready(...)`
- the client does not auto-ready on connect
- `rpc_game_started(...)` moves the client out of waiting room into the next
  placeholder state
- the code path remains small enough to explain without introducing a large
  general-purpose client-state framework

## Notes On Implementation

Keep the first waiting-room version intentionally small.

The main risk in this chunk is trying to solve all future match-state modeling
at once because `rpc_state_snapshot(...)` is broad. For this milestone, only
pull out the fields that the waiting room actually needs:

- room or game identifier
- local player id / index
- joined players
- ready states
- any obvious lifecycle flag that indicates the match is still waiting

If the snapshot shape is awkward, prefer a small adapter layer inside the
graphical client rather than leaking raw server dictionaries all over the UI.

## Manual Validation Checklist

1. Start the wrapper flow and reach `/launch.html`.
2. Confirm the client completes auth, join, and sync.
3. Confirm the generic session-check card hands off into a waiting-room scene.
4. Confirm the waiting room shows the local player and the known joined players.
5. Confirm the local player can click a ready action and send
   `rpc_player_ready(...)`.
6. Confirm the server broadcasts `rpc_player_ready_state(...)` and the waiting
   room updates visibly.
7. Confirm another player joining or readying updates the waiting-room state.
8. Confirm `rpc_game_started(...)` exits the waiting room into the next
   placeholder client state.

## Deferred Follow-Up

The next milestone after this one should likely be the first true gameplay root
that consumes:

- `rpc_board_state(...)`
- turn lifecycle broadcasts
- movement / landing events
- the broader reconnect snapshot model

The waiting room should become the boundary between session admission and
actual match play, not a second general-purpose networking layer.
