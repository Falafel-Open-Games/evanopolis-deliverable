# 016 - Durable Finished-Match Result Reporting

## Status

Planned.

## Context

The current architecture already splits responsibilities cleanly:

- `game-server` is authoritative for live gameplay, winner determination, and
  end-of-match tallies
- `rooms-api` is the durable HTTP surface keyed by `game_id`

What is still missing is a durable finished-match result record that survives
the lifetime of the live gameplay process and can be queried later by sponsors
or operators.

Without that persistence layer, sponsor settlement or prize payout workflows
would have to interrogate a live runtime or depend on transient in-memory match
state, which is not an acceptable external contract.

## Goal

Deliver the first durable finished-match result reporting path so that:

- `game-server` persists an authoritative finished result once a match ends
- `rooms-api` stores that finished result durably by `game_id`
- sponsor/operator clients can query whether a match finished and who won
- the reported result includes the minimum settlement and audit data needed for
  payout decisions

## Proposed Shape

- keep `game-server` as the only component that decides when a match is over
- have `game-server` write one authoritative final result payload to
  `rooms-api`
- have `rooms-api` expose a durable read API for finished-match lookup by
  `game_id`
- make the write path idempotent so retries cannot create divergent outcomes

## In Scope

- define the final result payload shape
- persist finished-match result records in `rooms-api`
- add an internal write path for `game-server`
- add a sponsor/operator read path keyed by `game_id`
- include winner, end reason, duration, and final standings/tallies
- add tests for successful persistence, duplicate writes, and result lookup

## Out Of Scope

- automatic payout execution
- sponsor authentication and authorization policy beyond what the first result
  lookup needs
- full analytics pipelines
- rewriting live gameplay persistence beyond the final result contract

## Acceptance Criteria

- when a match ends authoritatively, `game-server` can persist a finished-match
  result keyed by `game_id`
- the persisted record can be looked up later without consulting live
  in-memory gameplay state
- the record clearly identifies the winner and end reason
- the record includes enough final tally data for payout or audit workflows
- retries of the same authoritative result write remain safe and idempotent

## Notes On Implementation

Preferred boundary:

- `game-server` writes:
  - `game_id`
  - `started_at`
  - `finished_at`
  - `winner_index`
  - settlement-safe winner identity
  - `end_reason`
  - `player_count`
  - `turn_count`
  - `duration_seconds`
  - final standings / final balances / other settlement-relevant tallies
- `rooms-api` stores and serves that final record

Important constraint:

- `rooms-api` should not decide whether a live match has ended
- it should only persist and expose a result after `game-server` has already
  made the authoritative decision

## Manual Validation Checklist

1. Run a live match to completion.
2. Confirm `game-server` writes the final result record once the authoritative
   winner is decided.
3. Query the durable result by `game_id`.
4. Confirm the winner, end reason, and final tallies match the gameplay
   outcome.
5. Retry the same result write and confirm the stored result remains stable.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- sponsor-facing authentication and access control
- settlement status fields and payout tracking
- list/filter endpoints for recent finished matches
- richer audit metadata and analytics exports
