# 010 - Energy Allocation Slider

## Status

Done.

## Context

`009` completed the pending-action gameplay UI for buying, passing, toll
payment, and defeat acknowledgement. The next missing player-facing loop is the
energy allocation control that decides how owned-tile energy converts into
FIAT or BITCOIN over time.

The current HUD already has a slider placeholder, and the top bar already
surfaces player resources. What is missing is the authoritative rule contract
for:

- how a player sets SELL vs MINE allocation
- when that choice may be changed
- when the selected percentage takes effect
- how total owned energy stays visible across turns

## Goal

Deliver the first authoritative energy-allocation system in the graphical
client and server so that:

- each active player always sees their total owned energy in the top bar
- each active player has a persistent SELL vs MINE slider in the gameplay HUD
- each player starts from a neutral 50% SELL / 50% MINE allocation
- a player may change that slider at most once per turn, on any turn, not only
  their own
- the latest accepted percentage is applied at end-of-turn production time
- FIAT and BITCOIN earnings are calculated from the current owned-energy total
  plus the accepted allocation percentage

## Proposed Shape

- keep the server as the only source of truth for allocation percentage,
  eligibility, and production results
- make the slider a permanent gameplay control rather than a turn-local prompt
- treat allocation updates as lightweight match state, separate from movement
  and landing actions
- allow edits during any active turn, but rate-limit them to one accepted
  change per player per turn number
- apply production only when a turn resolves, using the latest accepted
  allocation value for each player

## In Scope

- add authoritative per-player energy allocation state to the match model and
  reconnect snapshot
- initialize new and restored players without explicit prior state to 50% SELL
  / 50% MINE
- define and wire an RPC for requesting an allocation change
- define and wire a broadcast/event for accepted allocation changes
- track whether a player has already changed allocation during the current turn
- calculate total owned energy from all currently owned properties
- keep total owned energy visible in the top bar for the local player
- enable the graphical slider for active players when they are eligible to
  change it
- apply the accepted percentage at the end of every resolved turn for each
  active player as production is calculated
- log or otherwise surface production/balance changes clearly enough for manual
  play testing

## Out Of Scope

- redesigning the top bar or HUD layout beyond what is needed for persistent
  slider visibility
- animation polish for slider interaction
- new resource types or new property classes
- market mechanics beyond the existing SELL and MINE split
- hidden or speculative client-side production estimates that diverge from
  server truth

## Acceptance Criteria

- the local player always sees their current total owned energy in the top bar
- the gameplay HUD shows a persistent SELL vs MINE slider for the local player
- the initial slider position is centered at 50% SELL / 50% MINE
- an active player can change the slider once during a turn, even when it is
  another player's turn
- a second change attempt by the same player during that same turn is rejected
  by the server
- reconnect/sync restores each player's current allocation percentage and
  per-turn edit eligibility correctly
- when a turn ends, the server applies production using the latest accepted
  allocation values
- resulting FIAT and BITCOIN balances update correctly in the graphical client

## Notes On Implementation

This milestone should stay grounded in an explicit authoritative contract.

Likely additions:

- snapshot fields for:
  - per-player allocation percentage
  - per-player last turn number changed
  - per-player current total owned energy, or enough tile ownership data for
    the client to derive it safely for display
- a request RPC such as:
  - `rpc_set_energy_allocation(game_id, player_id, sell_percent)`
- a broadcast such as:
  - `rpc_energy_allocation_changed(seq, player_index, sell_percent, turn_number)`
- production/balance events that make end-of-turn FIAT and BITCOIN gains
  visible to the client

The server should decide:

- the default initial allocation of 50% SELL / 50% MINE
- whether the player is active
- whether the player has already changed allocation this turn
- the normalized percentage value
- the exact FIAT and BITCOIN deltas produced at turn end

The client should present:

- the current allocation value
- whether the slider is editable right now
- the always-visible total owned energy for the local player

Per-player allocation state must be part of the authoritative snapshot so
reconnect, refresh, and resumed sessions restore the same production behavior
without relying on client memory.

## Manual Validation Checklist

1. Start a match with at least two active players.
2. Confirm the top bar shows the local player's current total owned energy.
3. Change the local slider during your own turn and confirm the server accepts
   it.
4. Change the local slider during another player's turn and confirm the server
   accepts it if you have not already changed it that turn.
5. Attempt a second change during the same turn and confirm it is rejected.
6. End a turn and confirm FIAT and BITCOIN production use the latest accepted
   allocation values.
7. Reconnect during a match and confirm allocation state and energy totals
   restore correctly.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- richer production history and end-of-turn summaries
- better affordance for slider lockout and next-turn availability
- per-player inspection of non-local allocation state
- endgame presentation once production pacing is validated in live play
