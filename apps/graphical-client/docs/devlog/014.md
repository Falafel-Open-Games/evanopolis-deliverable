# 014 - Variable Dice Count Selection

## Status

Planned.

## Context

`013` tracks authoritative pawn movement animation as a readability upgrade.
Separately, there is an explicitly requested gameplay change that adds a new
player decision before movement: choosing how many `d6` dice to roll.

Today, turn movement uses a fixed two-die roll. That keeps movement simple, but
it removes a potentially meaningful tactical choice. Allowing a player to pick
between one, two, or three `d6` dice before rolling would add controlled risk
management around toll avoidance, property targeting, and pacing.

This is a rules change, not just a presentation change, so it must remain
fully authoritative on the server.

## Goal

Deliver the first variable-dice-count gameplay system so that:

- the active player chooses `1`, `2`, or `3` `d6` dice before rolling
- the server authoritatively validates and applies the chosen dice count
- movement, landing resolution, and downstream actions still follow the current
  authoritative turn flow
- the gameplay UI makes the pre-roll choice explicit and readable

## Proposed Shape

- treat dice-count choice as part of the authoritative turn action state
- allow the active player to choose the dice count only before the roll for the
  current turn
- keep the server as the source of truth for allowed values, roll results, and
  resulting movement
- preserve the existing roll -> move -> resolve landing flow after the choice
  is made
- expose the chosen dice count clearly in the HUD and dice presentation

## In Scope

- add authoritative support for choosing `1`, `2`, or `3` dice before a roll
- validate that only the current active player can choose and use the value
- define the default dice count for a turn if the player does not change it
- update the gameplay HUD so the active player can make the choice clearly
- update client/server events and sync state so reconnect restores the current
  pre-roll dice-count state correctly
- keep movement and landing rules authoritative after the roll is resolved
- add focused tests for allowed choice, rejected choice, reconnect restore, and
  roll application

## Out Of Scope

- changing board topology or toll formulas
- adding non-`d6` dice types
- speculative client-side roll previews
- redesigning the full HUD beyond what the choice requires
- broader probability or balance overhaul beyond this mechanic itself

## Acceptance Criteria

- before rolling, the active player can choose `1`, `2`, or `3` dice
- the server rejects invalid dice-count requests or wrong-turn requests
- the next roll uses the chosen number of dice authoritatively
- the gameplay UI makes the current dice-count choice visible before the roll
- reconnect/snapshot restore preserves the correct dice-count state for the
  in-progress turn
- automated coverage exists for valid choice, rejected choice, and roll
  behavior using each supported dice count

## Notes On Implementation

This milestone touches both rules and UI, but the rule boundary should stay
clean:

- `apps/game-server/` owns:
  - allowed dice-count values
  - per-turn dice-count state
  - request validation
  - roll resolution and movement distance
  - reconnect snapshot fields
- `apps/graphical-client/` owns:
  - exposing the choice before roll
  - presenting the chosen count clearly
  - reflecting authoritative updates and reconnect restore

Key design questions to resolve during implementation:

- what is the per-turn default: keep `2` dice unless changed, or require an
  explicit choice every turn
- whether the choice can be changed multiple times before rolling, as long as
  no roll has happened yet
- how the dice presentation should render one-die and three-die rolls while
  preserving readability

The safest initial rule is:

- default each turn to `2` dice
- allow the active player to change the count freely until the roll is sent
- lock the choice once the authoritative roll occurs

That keeps the mechanic readable without adding unnecessary friction.

## Manual Validation Checklist

1. Start a live match in the graphical client.
2. On an active turn, choose `1` die and roll.
3. Confirm the server applies a single-die movement result correctly.
4. On a later active turn, choose `3` dice and roll.
5. Confirm the server applies the full three-die total correctly.
6. Confirm a non-current player cannot change or use the dice-count choice.
7. Reconnect before rolling and confirm the chosen count restores correctly.
8. Reconnect after rolling and confirm the turn remains in a correct
   post-roll state.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- clearer probability/risk messaging for each dice-count choice
- stronger visual treatment for one-die versus three-die rolls
- balance tuning if the new movement control changes match pacing too much
