# 014 - Scripted Dice Roll Presentation

## Status

Done.

## Context

`013` tracks pawn movement animation, and the current gameplay flow already
shows authoritative dice values in the center board. The next readability gap
is the transition between "roll requested" and "pawn starts moving."

Today, dice values update authoritatively, but the presentation can still feel
too abrupt when the visual result appears without a clear sense of motion. A
small scripted dice animation would make the roll easier to read before pawn
movement begins.

This does not require physics simulation. The server already decides the dice
results. The client only needs a short, illustrative presentation that lands on
the correct authoritative faces.

## Goal

Deliver the first scripted dice-roll presentation so that:

- players see a readable dice motion before pawn movement begins
- each die lands on the authoritative server-provided result
- the animation remains simple, deterministic, and non-physics-based
- reconnect and authoritative corrections can still interrupt or replace the
  presentation safely

## Proposed Shape

- animate each die with a short hop above the table
- rotate from the current orientation toward the final authoritative face while
  airborne
- add a small number of extra full spins so the movement feels deliberate
- land exactly on the final authoritative orientation before gameplay
  continues
- start pawn movement only after the dice presentation completes

## In Scope

- add a lightweight procedural dice animation in the gameplay client
- keep final die faces authoritative
- sequence pawn movement after the dice presentation instead of overlapping it
- allow later authoritative state updates to cancel or replace in-flight dice
  animation safely
- add a manual validation checklist for live gameplay readability

## Out Of Scope

- physics-based dice simulation
- random local dice outcomes
- changing gameplay rules around rolling
- broad cinematic camera work
- sound design beyond what a later slice may add

## Acceptance Criteria

- when the server reports a roll, the dice visibly hop and rotate before
  settling
- each die lands on the correct authoritative face
- pawn movement starts after the dice presentation finishes
- reconnect/snapshot restore can still place the dice correctly without
  depending on replaying the animation
- the presentation improves roll readability without introducing rule drift

## Notes On Implementation

This milestone should remain presentation-only.

The server owns:

- die values
- roll timing
- movement distance
- landing resolution

The client owns:

- short hop timing
- spin styling
- sequencing between dice presentation and pawn movement

A safe initial implementation is:

- fixed hop duration
- fixed hop height
- one or more extra full rotations layered on top of the current-to-target
  face reorientation
- immediate snap when reconnect/resync or other authoritative correction
  requires it

Prefer a deterministic procedural tween over rigid-body physics.

## Manual Validation Checklist

1. Start a live match in the graphical client.
2. Roll and confirm both dice visibly hop before settling.
3. Confirm each die lands on the authoritative rolled face.
4. Confirm pawn movement begins only after the dice presentation ends.
5. Confirm repeated rolls remain readable and do not leave dice in a broken
   intermediate pose.
6. Reconnect or resync during a match and confirm dice state restores cleanly.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- better timing variation between the two dice
- sound synced to takeoff and landing
- camera emphasis during the roll
- stronger visual linkage between dice total and pawn movement
