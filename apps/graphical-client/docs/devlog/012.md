# 012 - In-Match Disconnect Feedback And Recovery

## Status

Planned.

## Context

`011` completed the center-board dice presentation. The next reliability gap is
the in-match WebSocket/session experience when the live server connection is
lost during gameplay.

Today, if the transport drops during an active match, players often receive no
clear feedback in the gameplay UI. Actions may stop working, turn flow appears
frozen, and the only clue may be a browser-console RPC or WebSocket error. In
practice, that pushes players toward a full page refresh just to recover.

That is not good enough for real browser play. The graphical client should stay
resilient during temporary disconnects, keep the player informed, and attempt
to recover without forcing a manual reload.

## Goal

Deliver the first reconnect-aware in-match session UX so that:

- players get immediate visible feedback when the live server connection is lost
- gameplay does not silently fail or appear arbitrarily frozen
- the client attempts reconnect/sync recovery automatically when possible
- successful recovery restores the authoritative match view without a page
  refresh
- unrecoverable failure still leaves a clear player-facing explanation

## Proposed Shape

- treat transport loss during gameplay as a first-class session state
- surface a visible gameplay message when the socket disconnects or reconnect
  attempts are underway
- append concise event-log entries that explain what happened in plain language
- keep reconnect driven by the existing authoritative sync path instead of
  inventing local game-state recovery
- restore the match from server snapshot/sync once transport is healthy again

## In Scope

- detect in-match transport loss in the graphical client
- add player-facing gameplay feedback for:
  - connection lost
  - reconnecting
  - reconnected / resynced
  - unrecoverable disconnect failure
- retry connection automatically for a bounded number of attempts or until the
  current reconnect policy says to stop
- request authoritative sync again after reconnect succeeds
- ensure the gameplay scene remains usable after successful recovery without a
  full page refresh
- add clear manual validation notes for live browser testing

## Out Of Scope

- redesigning the auth/join bootstrap flow outside what reconnect reuse needs
- speculative offline mode or local prediction during disconnect
- complex multiplayer pause semantics
- full transport abstraction redesign if a smaller reconnect-state addition is
  enough
- non-essential visual polish beyond clear readability

## Acceptance Criteria

- when the gameplay socket/server connection drops, the player sees immediate
  on-screen feedback
- the event log records a readable explanation for the disconnect and recovery
  state
- the client automatically attempts reconnect without requiring a page refresh
- when reconnect succeeds, the client re-syncs authoritative state and gameplay
  resumes
- if reconnect fails permanently, the client still shows a clear player-facing
  error instead of silently freezing

## Notes On Implementation

This milestone should stay anchored to the existing session/runtime boundary:

- `SessionTransport`
- `SessionCheck`
- authoritative `rpc_state_snapshot(...)`
- authoritative `rpc_sync_complete(...)`

The main design constraint is that reconnect should remain server-authoritative.
The client should not try to preserve or invent local gameplay progress beyond
what the server re-sends during sync.

The simplest good boundary is:

- gameplay notices transport loss
- gameplay enters a reconnecting state with visible feedback
- transport reconnects
- session re-auths / re-joins as needed
- sync snapshot restores the current match

If some reconnect steps need broader session refactoring, keep the first slice
focused on player-visible correctness and safe authoritative recovery rather
than over-abstracting the transport stack.

## Manual Validation Checklist

1. Start a live match in the graphical client.
2. Disconnect the game server or block the socket during an active turn.
3. Confirm the player sees an immediate gameplay-visible disconnect message.
4. Confirm the event log records a readable connection-loss message.
5. Restore the server/connection and confirm the client attempts reconnect
   automatically.
6. Confirm the client re-syncs into the live match without a full page refresh.
7. Confirm actions work again after recovery.
8. Repeat with an unrecoverable failure and confirm the player still gets a
   clear terminal error state instead of a silent freeze.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- richer reconnect overlay presentation
- explicit countdown/backoff messaging
- reconnect-safe action pending indicators
- better differentiation between auth failure, room rejection, and pure socket
  loss
- optional wrapper-level awareness of active reconnect state
