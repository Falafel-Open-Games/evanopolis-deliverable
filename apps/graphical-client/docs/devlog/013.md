# 013 - Authoritative Pawn Movement Animation

## Status

Planned.

## Context

`012` closed the first reconnect-aware in-match recovery slice. The next
readability gap in live gameplay is pawn movement itself.

Today, pawn positions update authoritatively, but the graphical client mainly
jumps pawns to their latest tile instead of clearly showing movement across the
board. In real matches that makes turn resolution harder to follow, especially
when validating dice, passing start, toll landings, and reconnect-restored
state.

This is not a rules problem. It is a presentation gap that now affects match
comprehension during manual browser testing.

## Goal

Deliver the first authoritative pawn movement animation layer so that:

- players can visually follow pawn travel from one tile to the next
- dice results and resulting movement feel legible in live play
- the final pawn position always remains server-authoritative
- reconnect/snapshot sync still restores immediately and safely

## Proposed Shape

- animate movement only from authoritative position updates already emitted by
  the session/runtime boundary
- treat the authoritative destination tile as the source of truth
- keep animation lightweight and deterministic on the client
- snap immediately to the authoritative state when reconnect, resync, or other
  corrective updates require it
- avoid inventing local game rules, fake extra movement, or speculative path
  prediction

## In Scope

- animate pawn travel when a live authoritative tile-position update moves a
  pawn from one tile to another
- make multi-step movement readable enough for dice-driven turns
- preserve correct end position, owner visuals, and landing readability
- ensure reconnect/snapshot restore can still place pawns immediately without
  waiting for a replay animation
- add a clear manual validation checklist for live match testing

## Out Of Scope

- changing movement rules or board topology
- client-side path prediction beyond authoritative tile progression already
  known to the client
- cinematic camera work or broad animation polish
- adding new gameplay mechanics tied to movement
- replay systems or historical turn scrubbing

## Acceptance Criteria

- when a pawn moves during live gameplay, the player can see it travel across
  the board instead of only teleporting to the final tile
- the final resting tile still matches the authoritative server state
- passing start or moving across multiple tiles remains understandable in the
  animation
- reconnect/snapshot restore still places pawns correctly without relying on
  replaying prior movement
- the animation does not block later authoritative corrections from the server

## Notes On Implementation

This milestone should stay anchored to the current gameplay sync path:

- `SessionCheck`
- `AppMainView`
- `GameRoot`
- `PawnCollection`
- pawn scene/controller scripts under `scripts/game/pawns/`

The most important design constraint is that movement remains presentation
only.

The server already decides:

- whose turn is active
- what dice were rolled
- what the authoritative destination tile is
- whether the player passed start
- what landing outcome happens after movement

The client should only decide:

- how to interpolate or step between authoritative tile positions for
  readability
- when to skip animation and snap directly because reconnect/resync or other
  authoritative correction requires it

If the current client path already has enough information to animate passed
tiles cleanly, use it. If not, prefer a smaller but reliable animation over
inventing speculative movement state.

## Manual Validation Checklist

1. Start a live match in the graphical client.
2. Roll and confirm the active pawn visibly moves across the correct number of
   tiles.
3. Confirm the final pawn tile matches the authoritative gameplay result.
4. Confirm landings that trigger buy, toll, or end-turn actions remain easy to
   understand while the movement animation plays.
5. Confirm multi-tile movement that passes start remains readable.
6. Reconnect or resync during a match and confirm pawn placement restores
   correctly without broken in-between animation state.
7. Confirm later authoritative updates can still correct pawn positions
   cleanly if needed.

## Deferred Follow-Up

After this milestone, later slices can focus on:

- better pacing and easing for pawn travel
- camera framing that emphasizes the active movement
- stronger sync between dice presentation and pawn motion
- richer landing feedback once movement readability is reliable
