# 006 - 3D Table Foundation

## Status

Completed.

## Context

The waiting room and identity setup now feed enough authoritative player data
into the graphical client to support the first real in-match presentation
layer:

- player name, icon, and color are stable
- player colors are canonical across the room
- the active game scene can consume known player slots
- the board art direction has a usable first imported blockout

What this milestone actually needed was narrower than the original ambition.
The useful first step was not "full playable 3D gameplay." It was the
foundation that makes later gameplay integration straightforward:

- assemble the 3D table scene
- prove the board composition and camera framing
- expose a real pawn component and scene API
- place colored pawns on the board in stable default positions
- keep the implementation aligned with the authoritative server model

This is the table-foundation pass, not the full gameplay-integration pass.

## Goal

Deliver the first stable 3D game-table foundation that:

- establishes the board, table, and camera composition in the active game scene
- renders the simplified 18-tile hex ring in world space
- represents players as reusable colored pawns
- uses the waiting-room identity color contract consistently
- provides a clean base for later authoritative movement and tile-state updates

## Proposed Shape

- keep the server authoritative for actual match state
- treat imported Blender scenes as source data, not the long-term runtime API
- define a reusable pawn component that can be instantiated and configured at
  runtime
- keep the initial pawn placement rule simple and data-driven from the
  exported board/pawn markers
- standardize the canonical player color sequence before layering on more
  gameplay behavior
- prove scene composition and readability before integrating more action/UI state

## In Scope

- 3D table blockout for the active game scene
- camera framing and scene composition for the board
- simplified 18-tile ring present in the gameplay root
- reusable pawn runtime/component API
- pawn color mapping from waiting-room identity to in-match albedo
- canonical color ordering across identity editor, HUD, and pawns
- default per-color spawn placement from exported pawn markers snapped to
  board tiles
- documentation for the pawn API and current placement contract

## Out Of Scope

- full authoritative pawn movement during turn resolution
- buy / pass / toll prompt UI
- turn-state, landing, or reward presentation polish
- ownership stack visuals and tile mutation behavior
- reconnect-safe gameplay state installation beyond the current foundation hooks
- animation polish for movement, landing, or camera response
- any new gameplay mechanics or rule redesign

## Acceptance Criteria

- the client can enter a visible 3D game table after the waiting room
- the table composition is readable and anchored by a strong camera
- the 18-tile ring is visible and legible in the active game scene
- each player can be represented by a pawn using the canonical identity color
- pawn instancing and placement use a documented runtime API rather than
  hard-coded scene state
- the result is a stable base for later gameplay-event integration

## Notes On Implementation

The delivered implementation keeps the scope intentionally narrow:

- the gameplay root now owns a board presentation and pawn layer
- `Pawn` is a reusable runtime piece with explicit `player_index`, `color_id`,
  and board-transform setters
- `PawnCollection` is responsible for creating pawns, syncing known player
  slots, and exposing default spawn transforms
- the imported pawn GLB is treated as source data for geometry and color start
  markers, not as the public gameplay API
- canonical `color_id` order is now shared across the identity editor, HUD,
  and pawn presentation
- `0`: ice
- `1`: green
- `2`: sky
- `3`: red
- `4`: purple
- `5`: gold
- default spawn positions currently come from the exported pawn markers and are
  snapped onto the actual tile ring geometry

This keeps the first 3D milestone honest: it delivers the board foundation that
exists today, without pretending the full turn/landing flow is already
integrated.

## Manual Validation Checklist

1. Start from a waiting room with at least two players.
2. Enter the game scene and confirm the 3D table composition appears.
3. Confirm the camera frames the board cleanly.
4. Confirm the 18-tile ring is visible in the expected order.
5. Confirm each pawn uses the canonical player color.
6. Confirm each pawn appears on the expected default start tile for its color.
7. Confirm the table remains readable with the current gameplay HUD.
