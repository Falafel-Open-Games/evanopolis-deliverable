# 006 - 3D Gameplay Blockout

## Status

Planned.

## Context

The pre-match side of the graphical client is now in a good state:

- the waiting room is authoritative and readable
- the identity editor is stable
- colors are reserved consistently
- icons and pawns are tied to player identity
- the game-server snapshot already carries the information the client needs

That means the next useful chunk is not more lobby polish. It is the first
real 3D gameplay surface.

This stage should be treated as the foundation pass for the board:

- assemble the 3D table
- place the board loop in world space
- prove the camera and composition
- represent players with colored pawns
- keep the existing match flow intact

The visual bar still matters, but this is a blockout-and-integration milestone
first. The game should become playable and legible before it becomes pretty.

We also have a new direction reference from the tabletop mockup and the hand
sketches:

- a strong central board presentation
- clear side panels for player and property information
- a live event/status banner
- a readable buy/pass/turn feedback layer
- a diorama-like table surface with raised, sculpted tile cards
- a strict hexagon ring as the movement path
- a simplified 18-tile board with 3 tiles per side
- no special corner tiles in the first playable version
- rectangular landscape cards arranged along each side, with one center card
  and flanking left/right cards per side segment

Those ideas should guide the 3D work, but they do not need to be fully
polished in the first pass.

## Goal

Deliver the first playable 3D game surface that:

- establishes the table, camera, and lighting
- lays out the simplified hexagon loop of 18 tiles in world space
- shows the board as cards or cards-like tile pieces
- places each pawn on the board using the player’s chosen color
- preserves the existing match flow and core gameplay rules
- provides a readable base for later visual polish

## Proposed Shape

- keep the server authoritative for game state
- treat the board as a 3D tabletop presentation, not a new rules system
- build the loop from the existing city / tile data rather than inventing new
  board data
- simplify the movement ring to 18 tiles total, 3 per side, with no special
  corner tiles for the first release
- reuse the player identity color as the pawn color on the table
- keep movement, buying, tolls, and turns on the existing gameplay contract
- prefer clear, readable game surfaces over decorative complexity
- use temporary proxy geometry and placeholder materials first, then swap in
  final art once the layout is proven

## In Scope

- 3D table blockout for the active game scene
- camera framing and scene composition
- hex loop layout and tile placement in 3D
- city card or tile-card rendering for the current board set
- simplified 18-tile ring layout with 3 cards per side
- pawn placement and pawn color mapping from waiting-room identity
- basic gameplay readability for the first playable board view
- preserving the existing turn / roll / buy / toll flow
- adapting the current board to the revised city roster and energy source set
- keeping the client responsive and understandable during play
- placeholder UI surfaces for turn, reward, and landing feedback

## Out Of Scope

- a full visual redesign of the gameplay rules
- new gameplay mechanics
- custom board editor tools
- arbitrary tile creation
- profile systems or identity expansion
- final animation polish for every board action
- extra rule changes beyond what is needed to make the board playable
- final art polish before the table layout is proven

## Acceptance Criteria

- the client can enter a visible 3D game table after the waiting room
- the table composition is readable and anchored by a strong camera
- the board shows the simplified hexagon loop of 18 city cards clearly enough
  to play
- each player’s pawn appears with the correct color
- the board remains readable with the existing gameplay HUD and match flow
- basic gameplay still works on top of the new board presentation
- the result feels like the same game, not a different one

## Notes On Implementation

The safest approach is to keep the current authoritative match flow intact and
replace the gameplay surface around it.

That likely means:

- blocking out the scene with proxy meshes and placeholder materials first
- locking the camera and composition before spending time on final art
- assembling the 18-tile hex loop from existing board data and tile ordering
- mapping each tile to its landscape-oriented city card presentation
- rendering player pawns as identity-linked table pieces
- preserving the server snapshot as the source of truth for board state
- introducing only the minimum new client-side glue needed to render the
  table cleanly

The visual goal is a board that feels like a real tabletop game laid out on a
table, not a generic strategy grid. The technical goal is to get that into the
game loop in a way that can still be iterated quickly.

## Manual Validation Checklist

1. Start from a waiting room with at least two players.
2. Enter the game scene and confirm the 3D table composition appears.
3. Confirm the camera frames the board and side information cleanly.
4. Confirm the hex loop is readable and the 18 tiles are placed in the expected
   order.
5. Confirm each pawn uses the correct player color.
6. Confirm movement and turn flow still work on top of the new board view.
7. Confirm the revised city cards and energy sources are represented clearly.
8. Confirm the table still feels like the same game, even in placeholder form.

## Deferred Follow-Up

After this milestone, the next work should focus on turning the blockout into a
fully polished game scene:

- richer movement / landing presentation
- card and property interaction polish
- turn-state readability
- board action feedback
- broader gameplay animations and camera treatment
- final materials, props, and lighting once the layout is proven
- potential expansion back toward richer corner or landmark tiles only after
  the 18-tile core loop is stable

If the table itself needs another structural pass after the first integration,
that should become a separate follow-up rather than a reason to expand this
milestone into a design dump.
