# 005 - Waiting Room Identity Setup

## Status

Planned.

## Context

The waiting room already acts as the first real player-facing scene after the
session handshake. The next useful improvement is to let players configure a
small waiting-room identity before match start:

- short name
- one of a fixed set of icons
- one of a fixed set of colors

This is not a profile system and not gameplay. It is a lightweight pre-match
identity setup that makes the waiting room feel alive while the room fills.

Color is special: it is the pawn color and must be unique per player in the
room. If one player reserves a color, other clients should see that swatch as
disabled until it becomes available again. Icons do not need uniqueness; the
same icon can be reused by multiple players.

## Goal

Deliver a waiting-room identity setup flow that:

- lets the local player edit a short name
- lets the local player pick one icon from the fixed icon set
- lets the local player pick one color from the fixed color set
- prevents multiple players from reserving the same color at the same time
- keeps icon choice non-exclusive
- reflects identity state consistently across all connected clients
- keeps the waiting room readable while identity data is still being edited

## Proposed Shape

- keep the waiting room as the visible player-facing screen
- add a small identity editor area inside the waiting room
- treat identity values as per-player waiting-room state on the server
- extend the authoritative sync snapshot so each seat can carry identity data
- expose color availability to the UI so occupied colors render disabled
- allow icon choices to remain independently selectable even if already used
- keep the local ready action separate from identity editing

## In Scope

- short-name editing for the local player
- icon selection from the existing fixed icon set
- color selection from the existing fixed color set
- server-authoritative color reservation and release
- snapshot/broadcast updates that keep other clients in sync
- disabled color swatches when another player owns that color
- duplicate icon choices across players
- waiting-room UI updates that show the chosen identity clearly
- keeping the implementation compatible with the current waiting-room flow

## Out Of Scope

- full account/profile persistence
- arbitrary free-form avatar creation
- new gameplay mechanics
- board state or turn state changes
- broader match-start rules
- custom player art uploads
- any change that would turn identity setup into a separate profile system

## Acceptance Criteria

- the local player can change name, icon, and color from the waiting room
- color choices are exclusive across the room and visually disabled for others
- icon choices can be reused by multiple players
- identity updates appear on all connected clients through the authoritative
  waiting-room snapshot or equivalent broadcasts
- the waiting room remains readable and does not become a generic profile UI
- the identity editor fits the existing 640x360 target cleanly

## Notes On Implementation

The safest implementation is to keep identity metadata in the game-server
session state and propagate it through the sync snapshot.

That means the next code change will likely need:

- a small RPC/API review on the game-server side
- a snapshot field update for each player/seat
- a waiting-room UI update for disabled colors and selected icons

If a player changes color, the UI should immediately reflect the reservation
change once the server broadcasts the updated authoritative state. The same is
true if a player leaves and the color becomes available again.

## Manual Validation Checklist

1. Join a waiting room with at least two clients.
2. Change the local short name.
3. Pick an icon and confirm another client can pick the same icon.
4. Pick a color and confirm other clients see that color disabled.
5. Change away from that color and confirm it becomes available again.
6. Confirm the local identity choice persists in the waiting-room display.
7. Confirm the waiting room still updates ready state normally after identity
   changes.

## Deferred Follow-Up

The next milestone after this one should likely be the first gameplay root that
consumes:

- `rpc_board_state(...)`
- turn lifecycle broadcasts
- movement / landing events
- the broader reconnect snapshot model

If we later add a creator-only early-start action or an unready toggle, those
should remain separate follow-ups on top of the waiting-room identity system,
not be folded into this task.
