NEW RULESET SUMMARY FROM docs/game_rules.txt

Core goal:

- First player to reach 20 BITCOIN wins.

Core resources:

- FIAT
- ENERGY
- BITCOIN

Core resource loop:

- FIAT buys tiles.
- Owned tiles generate ENERGY.
- ENERGY is split between SELL and MINE.
- SELL generates FIAT.
- MINE generates BITCOIN.

Board structure:

- Hexagonal ring
- 6 sides
- 3 tiles per side
- 18 total tiles

Cities / energy types named in the new rules:

- Patagonia / Wind
- Ciudad del Este / Hydro
- El Salvador / Geothermal
- Angra dos Reis / Nuclear
- Atacama / Solar
- Irkutsk or Iceland / Cold Geothermal

Turn loop:

1. Roll dice
2. Move pawn
3. Resolve landing tile
4. End turn
5. Apply ENERGY allocation
6. Generate FIAT and BITCOIN

Landing logic:

- Unowned tile: buy option
- Other player owned tile: pay toll
- Self-owned tile: no penalty

Persistent strategic choice:

- Each player keeps an ENERGY split between SELL and MINE.
- SELL + MINE must always equal 100 percent.
- Allocation persists until changed.

Explicit V1 exclusions:

- no auctions
- no mortgage system
- no jail mechanics
- no complex trading
- no card system
- no multi-layer upgrades beyond 2-layer ownership presentation


HIGH-LEVEL DELTA: OLD RUNTIME VS NEW RULESET

The current runtime is built around a different economy and a different board
interaction model.

Major old-runtime mechanics that do not belong in the new rules:

- incident tiles
- bear and bull card decks
- inspection / jail-like mechanics
- inspection vouchers
- miner batches per property
- BTC rewards triggered by landing on owned property
- inflation cycles tied to laps
- property price scaling by cycle
- the old six-city set including Caracas, Assuncion, Minsk, Rockdale
- current board size default of 24 instead of the new 18-tile plan

Old doc text that should no longer drive implementation:

- "you win if opponents go broke"
- "you are the biggest miner after the length of the game"

Those win conditions appear in graphical-client design docs, but are not part
of the new authority and are not fully implemented in current runtime code
either.

CURRENT CONTRADICTIONS TO RESOLVE DURING THE REFACTOR

1. Runtime vs design authority

- The runtime still runs old rules.
- docs/game_rules.txt defines the intended new rules.

2. Win condition mismatch across repo

- Runtime code currently implements BTC-goal win.
- Some design docs still mention two extra win conditions that should be
  removed unless intentionally revived.

3. Board mismatch

- Current runtime board is 24+ tiles and old city names.
- New rules specify 18 tiles and a different city set.

4. Resource model mismatch

- Current runtime resource model is mostly fiat + bitcoin + miners.
- New rules require fiat + energy + bitcoin, with persistent allocation.

5. UI mismatch

- New gameplay mockup expects ENERGY split controls and production preview.
- Current runtime RPC and snapshot shape do not support that model yet.


SUGGESTED REFACTOR ORDER

1. Freeze authority

- Treat docs/game_rules.txt as the only gameplay-rules source for new work.
- Treat the gameplay mockup HTML as the main visual reference for gameplay UI.

2. Replace server rules core

- Rewrite apps/game-server/scripts/match.gd around the new economy loop.
- Rewrite economy_v0.gd or replace it with a new rules module.
- Update player and game state to carry ENERGY and allocation state.

3. Replace board model

- Rebuild board generation for 18 tiles and new cities.
- Remove incident and inspection tile logic if no longer present.

4. Replace RPC contract

- remove old gameplay-only RPCs that no longer exist
- add allocation update / production result events as needed
- update reconnect snapshot shape

5. Rewrite tests from rules authority

- start with failing tests for the new rule loop
- delete or replace incident / inspection / miner-batch tests
- update reconnect and snapshot tests after new state shape is stable

6. Update graphical-client gameplay layer

- keep session_check as the pre-game handshake layer
- build gameplay UI from the new mockup and the new RPC contract

7. Clean stale docs and stale copy

- update or remove old win-condition text
- update waiting-room wording if needed
- mark old docs as obsolete if they remain for history
